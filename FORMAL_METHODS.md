# Formal Methods Assurance — Honest Assessment

This document is a self-audit of Nucleus's formal verification claims.
It tries to be more critical of our work than any external reviewer would be.
If you find a gap this document doesn't disclose, please file an issue.

## What We Verify

### 1. Permission Lattice Algebra (Lean 4 — unbounded, kernel-checked)

**What's proved:**
- `CapabilityLevel` (3-element total order) is a `HeytingAlgebra` — full Mathlib instance
- `CapabilityLattice` (12-dimensional product) is a `HeytingAlgebra` — 26 theorems
- Function correspondence: the Aeneas-generated `meet`/`join`/`implies` compute the
  same result as the Mathlib lattice `⊓`/`⊔`/`⇨` — closing the loop between Rust and Lean
- Monotonicity: `(a ⊓ b) ≤ a` — permissions can only tighten (VC-001)
- Idempotency: `a ⊓ a = a` — applying the same policy twice is a no-op

**What's NOT proved:**
- The `PermissionLattice` (which wraps `CapabilityLattice` + path rules + budget + time)
  is not in Lean. Only the capability dimensions are verified algebraically.
- The Aeneas pipeline only translates `CapabilityLevel` and `CapabilityLattice`.
  `ExposureSet`, `Operation`, and `PureVerdict` are not yet translatable by Aeneas
  (it silently drops `bool`-field structs and 12-variant enums with `Option` returns).

**Correspondence gap:**
The Lean types (`PortcullisCore.Types`) are machine-generated by Aeneas from the
Rust source. The lattice FUNCTIONS (`CoreFuns.lean`) are hand-curated extracts from
the full Aeneas output (`Funs.lean`), because the full output includes trait impls
that don't compile under Lean 4.28. The curated functions are UNMODIFIED from Aeneas
output — the bodies are byte-for-byte identical — but we selected which functions
to include. CI verifies the Aeneas output hasn't drifted from the committed code.

### 2. Exposure Tracker (Lean 4 — unbounded, kernel-checked, hand-written model)

**What's proved:**
- Monoid laws: `union` is associative, commutative, idempotent with `empty` as identity
- Monotonicity: exposure count never decreases under `union`
- Once uninhabitable, always uninhabitable (under `union`)
- Uninhabitable ↔ count = 3 (sound and complete)
- `classify_operation` maps all 12 operations to the correct labels
- `should_gate` blocks exfil when completing/maintaining uninhabitable state
- `should_gate` never blocks non-exfil operations
- `should_gate` never blocks anything from empty state

**What's NOT proved:**
- These proofs are on a HAND-WRITTEN Lean model, not Aeneas-generated code.
  Aeneas cannot currently translate `ExposureSet` (struct with `bool` fields).
- Correspondence between the Lean model and the Rust source is enforced by
  `include_str!` compile-time tests that check discriminant values and field names,
  not by a machine-verified bisimulation.
- A developer could change the Rust logic without updating the Lean model.
  The `include_str!` tests catch structural drift (new fields, renamed variants)
  but not semantic drift (changed `match` arms with the same field names).

### 3. DecisionToken Linearity (Kani BMC — bounded)

**What's proved (5 harnesses, E1-E5):**
- E1: `decide()` produces a token iff verdict is `Allow` (symbolic over all 12 ops)
- E2: Denied operations never produce tokens (symbolic over all 12 ops)
- E3: Token operation and sequence match the decision
- E4: `issue_approved_token()` produces audited tokens with exposure tracking
- E5: `issue_approved_token()` bypass under deny policy is still audited

**What's NOT proved:**
- These are BOUNDED model checks, not full proofs. Kani explores all paths up to
  a bound but does not guarantee absence of bugs in deeper executions.
- `issue_approved_token()` is `pub` and bypasses the policy pipeline by design
  (for the `RequiresApproval` → external approval flow). It has safety documentation
  but no type-level enforcement of the precondition.
- The `_seal: ()` privacy relies on Rust's module system, not a cryptographic
  mechanism. A `#[cfg(test)]` or `unsafe` block within the `kernel` module could
  construct a token. This is standard Rust encapsulation, not formal sealing.

### 4. Constitutional Kernel (Kani BMC — bounded)

**What's proved (17 harnesses):**
- Budget escalation detection
- Capability non-escalation (bitmask encoding)
- I/O confinement
- Combined monotonicity
- Lattice subset transitivity
- Various symbolic BTreeSet proofs (nightly only)

**What's NOT proved:**
- These proofs operate on a simplified bitmask encoding, not the full `BTreeMap`-based
  policy. The bitmask proofs are sound for the encoded properties but don't cover
  the full policy evaluation path.
- 5 harnesses run per PR; 12 are nightly-only (symbolic BTreeSet proofs take 5-15 min each).

## What We DON'T Verify

These are important security properties that have NO formal verification:

1. **Sandbox isolation** — Firecracker VM boundary, seccomp BPF filters, filesystem
   sandboxing. Tested but not formally verified.

2. **Cryptographic attestation** — Ed25519 signing in the witness chain uses the
   `ed25519-dalek` library. We trust the library, not our own proof.

3. **Network policy enforcement** — `iptables` rules for agent network isolation.
   Tested, not verified.

4. **Time and budget checks** — `Kernel::decide()` checks session expiry and budget
   limits using `chrono` and `rust_decimal`. These are runtime checks with no formal
   model.

5. **Path and command restrictions** — File path matching and command allowlisting
   use string operations. No formal model of filesystem semantics.

6. **Concurrency** — The kernel is single-threaded per session. Multi-session
   coordination (executor pool, work queue) is tested but not formally modeled.

## Known Limitations & Honest Caveats

1. **Kani is bounded, not complete.** Kani proves properties hold for all inputs UP TO
   a symbolic bound. It is not a theorem prover. For the lattice properties (3-element
   type, 12-dimensional product), the bound covers the entire state space. For kernel
   decision logic with strings and paths, the bound is a practical limit.

2. **Lean proofs are on models, not machine code.** The Lean proofs verify properties
   of Lean functions. The Aeneas pipeline provides correspondence between Rust source
   and Lean functions, but this does not extend to the compiled binary. A compiler bug
   in `rustc` could violate proved properties. This is the standard "verified source,
   trusted compiler" gap shared by every verification project except CompCert and seL4.

3. **The exposure tracker Lean model is hand-written.** Unlike the lattice types
   (machine-generated by Aeneas), the exposure tracker model was written by hand to
   match the Rust source. Compile-time `include_str!` tests enforce structural
   correspondence but not semantic correspondence.

4. **No proof of the full `decide()` function.** The kernel's decision function calls
   the lattice (verified) and the exposure tracker (verified) but also checks time,
   budget, paths, commands, and isolation (not verified). A `decide_pure()` function
   has been extracted to `portcullis-core` as the next verification target.

5. **CI runs 10 of 67 Kani proofs per PR.** The remaining 57 run nightly. A PR could
   merge with a broken proof if it only breaks a nightly harness. The proof count
   regression gate catches deletion but not breakage of proofs not in the fast tier.

## Comparison to Other Projects

| Project | Scope | Proof depth | Our gap |
|---------|-------|-------------|---------|
| seL4 | Microkernel | Full functional correctness, binary verified | We verify algebra + decision logic, not the full kernel or binary |
| AWS Cedar | Authorization policy | Lean 4 policy evaluation | We verify the permission lattice structure, not full policy evaluation |
| HACL* | Cryptography | F* + C extraction | We trust ed25519-dalek, don't verify crypto ourselves |
| CompCert | C compiler | Coq, binary-level | We use the standard "trusted compiler" assumption |
| Aeneas (their own examples) | Rust → Lean | Borrow-checking soundness | We use Aeneas for types + functions, not full borrow-checking proof |

## How to Reproduce

```bash
# Lean 4 proofs (requires elan)
cd crates/portcullis-core/lean
lake build PortcullisCoreBridge   # HeytingAlgebra + function correspondence
lake build ExposureProofs          # Exposure tracker safety properties

# Kani BMC (requires kani toolchain)
cargo kani -p portcullis --solver cadical    # 50 harnesses
cargo kani -p ck-kernel --solver cadical     # 17 harnesses

# Verify no sorry (proof holes)
grep -rn 'sorry' crates/portcullis-core/lean/ --include='*.lean'
# Should return only comments, never proof terms

# Verify Aeneas output matches committed code
./scripts/aeneas-translate.sh
diff -r crates/portcullis-core/lean/generated/ <fresh-output>
```
